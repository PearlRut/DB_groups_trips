# שלב ה' - ממשק גרפי לבסיס הנתונים

## מערכת לניהול טיולים וקבוצות

בשלב זה נבנה ממשק גרפי למערכת ניהול טיולים וקבוצות.  
הממשק מאפשר לעבוד מול בסיס הנתונים דרך מסכים גרפיים ונוחים, לבצע פעולות CRUD, להפעיל דוחות ושאילתות, להפעיל פונקציה ופרוצדורה שנבנו בשלבים הקודמים, ולצפות בלוגים של שינויים במערכת.

האפליקציה נכתבה בשפת Python בעזרת Tkinter, ומתחברת למסד הנתונים PostgreSQL באמצעות psycopg2.

---

# הוראות הפעלה של האפליקציה

## 1. התקנת ספריות נדרשות

לפני ההרצה יש לוודא שמותקנות הספריות הבאות:

```bash
pip install psycopg2-binary
pip install python-dotenv
```

אם הפקודה `pip` לא עובדת, ניתן להריץ:

```bash
python -m pip install psycopg2-binary
python -m pip install python-dotenv
```

---

## 2. הגדרת קובץ התחברות למסד הנתונים

בפרויקט קיים קובץ `.env` שבו מוגדרים פרטי החיבור למסד הנתונים.

מבנה הקובץ:

```env
PHASE_E_DB_HOST=localhost
PHASE_E_DB_NAME=unified_db
PHASE_E_DB_USER=postgres
PHASE_E_DB_PASSWORD=your_password_here
PHASE_E_DB_PORT=5432
```

יש לוודא ששם בסיס הנתונים, המשתמש, הסיסמה והפורט מתאימים למחשב שבו מריצים את האפליקציה.

---

## 3. בדיקת חיבור למסד הנתונים

נכנסים לתיקיית האפליקציה:

```bash
cd Phase_E/app
```

מריצים:

```bash
python test_connection.py
```

אם החיבור תקין, תופיע הודעה שהחיבור הצליח ויוצגו מספר רשומות מטבלת הטיולים.

---

## 4. הפעלת האפליקציה

מתוך התיקייה:

```bash
Phase_E/app
```

מריצים:

```bash
python main.py
```

לאחר מכן ייפתח חלון המערכת.

---

# דרך העבודה והכלים שבהם השתמשנו

האפליקציה נבנתה באמצעות Python ו־Tkinter.  
Tkinter שימש לבניית המסכים הגרפיים, הטפסים, הכפתורים והטבלאות.

החיבור למסד הנתונים PostgreSQL מתבצע באמצעות הספרייה `psycopg2`.  
פרטי החיבור נטענים מתוך קובץ `.env` באמצעות `python-dotenv`, כדי לא לכתוב את פרטי החיבור ישירות בתוך הקוד.

מבנה הקוד חולק למספר קבצים כדי לשמור על סדר:

- `main.py` - קובץ ההפעלה הראשי של האפליקציה.
- `app_controller.py` - אחראי על מעבר בין מסכי המערכת.
- `db.py` - אחראי על החיבור למסד הנתונים והרצת שאילתות.
- `config.py` - קורא את פרטי החיבור מתוך קובץ `.env`.
- `ui_helpers.py` - מכיל פונקציות עזר לעיצוב כפתורים, כותרות וטבלאות.
- תיקיית `screens` - מכילה את מסכי המערכת, כאשר לכל מסך יש קובץ נפרד.

החלוקה למסכים נפרדים מאפשרת עבודה מסודרת וברורה יותר, ומקלה על תחזוקה והרחבה של המערכת.

# פעולות CRUD במערכת

במערכת קיימת אפשרות לבצע פעולות CRUD מול טבלאות בסיס הנתונים:

- **Create** - הוספת רשומה חדשה.
- **Read** - הצגת רשומות ושליפה לפי מפתח.
- **Update** - עדכון רשומה קיימת.
- **Delete** - מחיקת רשומה קיימת.

בזמן עדכון, המשתמש יכול להזין מפתח או לבחור רשומה מהטבלה.  
לאחר מכן המערכת טוענת את שאר השדות לטופס, ומשם ניתן לעדכן את הרשומה.

במקומות שבהם יש מפתחות זרים, הממשק מציג למשתמש שמות ונתונים ברורים במקום מספרי ID.  
לדוגמה:

- במסך טיולים מוצג שם המסלול במקום `route_id`.
- במסך טיולים מוצג שם סוג התחבורה במקום `transport_type_id`.
- במסך ציוד מוצג שם הספק במקום `supplierid`.
- במסך תחבורה מוצגים שם הספק ושם סוג התחבורה.
- במסכי השיבוץ מוצגים שמות טיולים, משתתפים, ציוד, מיקומים ותחבורה.

# דגשים להגנה

במהלך ההגנה ניתן להציג את הדברים הבאים:

1. המסך הראשי מאפשר גישה לכל מסכי המערכת.
2. קיימים מסכי CRUD לטבלאות בסיס הנתונים.
3. טבלאות קשר כמו `trip_participants`, `trip_equipment`, `location_trip` ו־`trip_transportation` קיבלו מסכים ייעודיים.
4. מפתחות זרים מוצגים בצורה ידידותית באמצעות שמות ולא רק באמצעות ID.
5. עדכון מתבצע לפי הדרישה: שליפה לפי מפתח, טעינת הנתונים לטופס, ואז עדכון.
6. מסך הדוחות מפעיל שתי שאילתות/Views משלב ב'.
7. מסך הדוחות מפעיל פונקציה ופרוצדורה משלב ד'.
8. ניתן לראות השפעה של פעולות, לדוגמה שיבוץ משתתפים משפיע על תפוסת טיול, והפעלת הפרוצדורה יכולה לעדכן סטטוס טיול.
9. טבלאות הלוג מוצגות לקריאה בלבד, משום שהן מיועדות לתיעוד שינויים.
